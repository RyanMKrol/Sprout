## Overview
On the Add Plants basket (Sources/Views/AddFlowView.swift) each basket row has a shuffle/'random' button (AddFlowView.swift:322 calls viewModel.reroll(entry); the button is Image(systemName: "shuffle") at line 324 with accessibilityLabel "New random name" at line 328). Today reroll(entry) (BasketAddViewModel.swift:148-151) ONLY reassigns the entry's nickname from the injected seedable RNG. The user wants a single shuffle tap to ALSO reroll the token GLYPH (the plant icon) and its token COLOUR, so one tap gives a fresh name + look together. The glyph is already a stored `var icon: PlantIcon` on BasketAddViewModel.Entry (BasketAddViewModel.swift:36-43) and the row renders it via FixedGlyphPlantToken(icon: entry.icon, duo: PlantTokenPalette.duo(for: entry.id), size: 34) at AddFlowView.swift:304. The colour is NOT stored: it is derived from the entry's UUID by PlantTokenPalette.duo(for: entry.id) (PlantToken.swift:29-32 hashes the UUID and picks one of PlantTokenPalette.duos — 6 duos). Because the entry's id is a `let` (immutable), the colour cannot be rerolled by mutating the id; the smallest deterministic mechanism is to store a chosen palette index on the Entry and render the basket token from it. This unit keeps the change BASKET-LOCAL (it does NOT persist the rerolled colour onto the created Plant — see the pending colour-mechanism question; if the owner wants persistence a follow-up will extend it to the Plant model + all render sites). Must reuse the existing injected seedable RNG (AnyRandomNumberGenerator / RandomNicknameProvider / SeededRandomNumberGenerator, BasketAddViewModel.swift:68-78) so DemoSeed and tests stay deterministic.
