# Worklog — T011 Check-in flow UI (soil / leaves / watered → recommendation)

## 2026-06-01 — built (RyanMKrol / autonomous builder)

Fresh start: clean `t011` branch off T010, no prior partial work.

### What I did
- **`Sources/ViewModels/CheckInViewModel.swift`** (new) — `@MainActor ObservableObject` driving
  the flow. Published inputs `soil`/`leaves`/`watered`; `load()` resolves the plant + its
  `CareProfile` (sets `canCheckIn`); `submit(now:)` builds the `CheckIn`, runs the pure
  `AdaptiveEngine` (T010), **persists** via `repository.addCheckIn` + writes the nudged
  `adj`/`lastWatered`/`nextDue` back via `repository.update`, then publishes a `Result`
  (recommendation + nextDue + didWater + newAdj). `Result.message` maps each `reason` to the
  design's per-row indication text. `now` injected for deterministic tests.
- **`Sources/Views/CheckInView.swift`** (new) — sheet with segmented Soil (Dry/Moist/Wet) and
  Leaves (Fine/Droopy) pickers, "I watered it" toggle, "Save check-in" (disabled when
  `!canCheckIn`), and a result form showing the recommendation icon/message + next-due. Degrades
  to an unavailable state on `loadFailed`.
- **Wiring (outside literal scope, minimal):** `ContentView` adds `makeCheckIn`; `PlantListView`
  threads it to the detail destination and routes the `SPROUT_SCREEN=checkin` deep-link (push
  first plant → detail auto-opens the sheet); `PlantDetailView` gains a "Check in" button + sheet
  and reloads on dismissal. Recorded in `docs/LIMITATIONS.md`.
- **`Tests/CheckInViewModelTests.swift`** (new, 16 tests) — end-to-end over a real in-memory
  `PlantRepository`: every soil/leaf combo → correct action/reason + exact `adj` nudge, didWater
  logic, next-due (watered → interval; skip/declined → `recheckDays`), check-in persisted to
  history, plant state written back, unknown-species `canCheckIn=false`, missing-plant
  `loadFailed`.
- Docs in lockstep: `TASKS.md` box ticked, `README.md` row → ✅, four `docs/LIMITATIONS.md` rows.
