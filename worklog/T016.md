# T016 — Feed weather into the schedule engine

## 2026-06-01 — built (done)

### What I did
- **Pure mapping (`Sources/Engine/WeatherFactor.swift`, new):** `WeatherFactor.factor(for:outdoor:)`
  maps a `WeatherForecast` to the `weatherFactor` the engine multiplies into the interval.
  Temperature-led: daily-**mean** temperature vs a neutral band `16–24 °C`, slope `±0.02`/°C,
  clamped to `[0.7, 1.3]` → **hot ⇒ `< 1`, cold ⇒ `> 1`**, mild ⇒ `1.0`. Optional precipitation
  term (`outdoor: true`) lengthens the interval for recent rain; default `outdoor: false` so
  houseplants ignore rain. Empty forecast ⇒ neutral.
- **`WeatherFactorService.factor(for:)`** now delegates to `WeatherFactor.factor(for:)` (was the
  T015 neutral placeholder), so the live `currentWeatherFactor()` path produces a real factor.
- **Surfacing in the "why" explanation:** already wired in T015 (`ScheduleExplanation.Cause`
  `.warmSpell`/`.coldSpell`, chosen when an off-neutral `weatherFactor` is the dominant mover).
  No engine change needed — verified by existing + new tests.
- **End-to-end app wiring:** `ContentView` holds a `weatherFactor` state, refreshed by a `.task`
  via `WeatherFactorService(CoreLocationProvider, OpenMeteoWeatherProvider)`, **gated on
  `AppSettings.weatherEnabled`** and **skipped under `-seedDemoData`** (fixed demo factor instead,
  so screenshots are deterministic and never trigger a CoreLocation prompt). The factor is passed
  into `PlantDetailViewModel` (explanation) and `CheckInViewModel` (recompute), both via a new
  defaulted `weatherFactor` parameter (neutral default ⇒ no test regressions).
- **DemoSeed:** added `DemoSeed.weatherFactor` (a fixed ~30 °C warm-spell forecast → 0.88) and made
  Lily weather-dominant (neutral `adj` + her existing check-in history) so the `detail` deep-link
  shows the weather story while still rendering history.
