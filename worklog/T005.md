**What I did (Scope: `Sources/Persistence/*`, `Tests/Persistence*`):**
- `Sources/Persistence/PlantRepository.swift` — the persistence-boundary **protocol** (imports
  only `Foundation`, no SwiftData): `allPlants()`, `plant(id:)`, `add(_:)`, `update(_:)`,
  `delete(id:)`, `addCheckIn(_:toPlant:)`, plus `PlantRepositoryError.notFound(UUID)`. All
  inputs/outputs are the pure domain value types — nothing SwiftData-shaped crosses this line.
- `Sources/Persistence/StoredModels.swift` — SwiftData `@Model` store types `StoredPlant` and
  `StoredCheckIn` with a cascade `@Relationship` (plant → check-ins), plus `init(domain:)` /
  `toDomain()` mapping both ways. `toDomain()` returns check-ins **sorted chronologically**.
  Enum columns store the domain `SoilMoisture`/`LeafState` directly.
- `Sources/Persistence/SwiftDataPlantRepository.swift` — the `PlantRepository` implementation
  over a `ModelContext` (CRUD + append check-in; `update` rewrites scalar fields only), and a
  `PlantStore` factory (`inMemory()` / `persistent()`) so callers/tests get a repository
  without importing SwiftData themselves.
- `Tests/PersistenceTests.swift` — 12 integration tests against a fresh **in-memory**
  `ModelContainer` (via `PlantStore.inMemory()`): round-trips a plant **with** check-ins;
  add/fetch with no check-ins; missing-plant → nil; `allPlants()` sorted; empty store; update
  scalars persists and keeps check-ins; update/delete/addCheckIn on a missing plant throw
  `.notFound`; delete cascades check-ins; check-ins returned chronologically even when appended
  out of order.
