# T218 — My Plants: swipe-to-delete + edit rework (drop species)

## 2026-06-02

**Done.** Built, tested, screenshotted, committed, pushed on branch `t218`.

### What I did
- **Swipe-to-delete on My Plants** (`PlantListView.swift`): added a destructive
  "Delete" button to each row's trailing `swipeActions`, alongside the existing "Edit".
  It calls a new `PlantListViewModel.delete(id:)` which routes through
  `repository.delete(id:)` and reloads the list.
- **Edit form narrowed to nickname + room** (`PlantEditView.swift`,
  `PlantEditViewModel.swift`): removed the Species section/picker from the view and the
  species-picker surface from the view model (`speciesQuery`, `select`, `isSelected`,
  `allSpecies`, `speciesResults`, `selectedSpecies`). `canSave` is now just a non-blank
  nickname; `save()` in edit mode updates nickname + room and **preserves species and all
  scheduling state** (`adj`/`lastWatered`/`nextDue`/`checkIns`). Species is loaded from
  the plant on open and never changed.
- **Tests** (`Tests/PlantEditViewModelTests.swift`): rewrote for the narrowed edit —
  canSave (nickname only), prefill of nickname/room, save preserves species + scheduling
  state, clearing the room, missing-plant load failure. Dropped the old species-picker /
  add-mode tests.
- **Screenshot deep-link**: added `SPROUT_SCREEN=edit` → first plant's edit sheet
  (`PlantListView.deepLinkIfRequested` + the `HomeView` deep-link router that pushes the
  list) so the narrowed form is screenshottable.

### Scope note
The task's `Scope:` listed only `PlantListView/PlantEditView/PlantEditViewModel` + the test.
Two adjacent files were touched out of necessity: `PlantListViewModel.swift` (the view's only
seam to the repository — the new `delete(id:)`), and `HomeView.swift` (one extra `case "edit"`
in the existing deep-link `switch`, needed for the screenshot). Both are minimal and
pattern-consistent.
