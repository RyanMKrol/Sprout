# T013 — Local watering notifications

## 2026-06-01 — built (done)

**What I did**
- Added `Sources/Notifications/NotificationScheduling.swift` — the `NotificationScheduling`
  protocol (async `requestAuthorization`, `scheduleReminder(for:)`, `cancelReminder(for:)`),
  the persistence-style boundary the app/view-models depend on instead of
  `UNUserNotificationCenter` directly.
- Added `Sources/Notifications/WateringNotificationScheduler.swift` — the production
  implementation. Defines a minimal `UserNotificationCenter` protocol whose signatures match
  `UNUserNotificationCenter`'s async API exactly (so `extension UNUserNotificationCenter:
  UserNotificationCenter {}` is empty), letting tests inject a stub. One pending reminder per
  plant keyed by `sprout.watering.<uuid>`; a `UNCalendarNotificationTrigger` fires on the
  `nextDue` day at `defaultReminderHour = 9`. `scheduleReminder` removes the plant's existing
  request first (reschedule = replace, never stack), requests authorization lazily on first
  use, and degrades to a no-op if denied. `cancelReminder` clears the plant's request.
- Added `Tests/NotificationSchedulingTests.swift` — 8 integration tests against a recording
  `StubNotificationCenter`: schedules for the right date/hour, mentions the plant, no reminder
  without `nextDue`, reschedule replaces (count stays 1, date moves), distinct plants get
  distinct reminders, cancel removes, denied auth schedules nothing, requestAuthorization
  returns the granted/denied decision.
