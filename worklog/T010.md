# T010 — Adaptive update from a check-in (pure)

## 2026-06-01

**Did:** Implemented the pure adaptive update per `docs/designs/adaptive-watering.md`.

- New `Sources/Engine/AdaptiveEngine.swift`:
  - `CheckInTiming` (early/onTime/late), derived from the plant's `nextDue` (or
    `lastWatered + interval`, else on-time) vs `checkIn.date`, with a ±20% interval band.
  - `WateringRecommendation` — a **structured** value (`Action` × `Reason` enum + `days`),
    one `Reason` case per row of the design table, so tests assert the *decision* not a string.
  - `CheckInUpdate` returns `(newAdj, recommendation, didWater, lastWatered, nextDue)`.
  - `AdaptiveEngine.update(profile:plant:checkIn:weatherFactor:)` applies the full decision
    table (droopy overrides; wet→skip+lengthen ×1.15; dry-early→shorten ×0.85; dry on-time
    hold / staysMoist ×0.90; moist hold / early ×1.05; droopy dry ×0.80 / wet ×1.20 overwater
    / moist ×0.95 monitor), clamps `adj` to `[0.5, 2.0]`, and recomputes the schedule:
    watered ⇒ `lastWatered = checkIn.date` + new `nextDue`; skip ⇒ `nextDue = checkIn.date + 3`.
  - Schedule recompute delegates to `ScheduleEngine` (T009), so `weatherFactor` + the adj clamp
    stay consistent.
- New `Tests/AdaptiveEngineTests.swift`: 24 tests covering **every** table row (incl. the three
  called-out cases: wet→skip+lengthen, dry-early→shorten, droopy+wet→lengthen+overwater), the
  `[0.5, 2.0]` clamp (both bounds), the water-vs-recheck schedule paths (incl. recommended-water-
  but-user-declined), timing classification + fallback, and weatherFactor flow-through.
